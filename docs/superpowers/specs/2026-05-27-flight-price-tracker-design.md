# 机票价格追踪与每日提醒 — 设计文档

- 日期:2026-05-27
- 状态:已通过 brainstorming,待用户最终评审 → 进入 writing-plans

## 1. 目标

每天在固定时间(约 Tijuana 时间 9 点)自动查询 TIJ→广州/深圳的往返机票价格,结合 Google Flights 的"价格洞察"和自建的历史价格,判断"现在该不该买",并通过 Telegram 推送给用户。从项目上线到用户真正买到票为止,**每天都发一条**,哪怕当天没有好价。

### 背景与硬约束

- 行程:**3 人往返**,从 **Tijuana (TIJ)** 出发到 **广州 (CAN) / 深圳 (SZX)**。
- 直飞优先,可接受转机(含中国境内转机)。TIJ→中国基本无直飞,必然有转机。
- 追踪的 4 对日期(返回 2/12 已弃用):
  | # | 出发 | 返回 |
  |---|------|------|
  | 1 | 2026-12-28 | 2027-02-16 |
  | 2 | 2026-12-28 | 2027-02-17 |
  | 3 | 2026-12-29 | 2027-02-16 |
  | 4 | 2026-12-29 | 2027-02-17 |
  日期仍可能调整 → 全部放配置,改配置即可。
- 货币:**MXN**(墨西哥比索)。
- 航司偏好:**海南航空(HU)**,但 TIJ 出发难有纯海航行程(见 §6 决策)。
- 用户 6–9 月大概不会购买,但仍要每天收到信息。

## 2. 关键技术决策

| 项目 | 决策 | 理由 |
|------|------|------|
| 价格 + "该不该买"信号 | **SerpAPI Google Flights 接口** | 唯一能直接返回 `price_insights`(偏低/正常/偏高 + 典型价格区间),自带"同期对比";免费额度够用 |
| 运行环境 | **GitHub Actions**(cron 定时) | 完全免费、云端运行,不依赖用户电脑开机 |
| 通知方式 | **Telegram 机器人** | 免费、手机推送、与 Actions 集成最简单 |
| 语言 | **Python** | 写这类脚本生态最顺手 |
| 历史数据存储 | 仓库内 **CSV**,Actions 自动 commit 回仓库 | 免费、带版本、可读、易画图,无需数据库 |

### 查询预算

- SerpAPI 免费额度:**250 次查询/月**(用户已在后台确认)。
- 每天查询数:**4 对日期 × 2 个城市 = 8 次/天**。
- 月用量:8 × 31 = **248 次/月**(最长月份),≤ 250,**刚好够**。
- SerpAPI 通常只对**成功的查询**计费,失败重试不计入额度,所以余量安全。
- "每天查几个组合"做成配置项;失败只重试不浪费成功额度,绝不超 250。

## 3. 架构与模块

整体流程(全部在云端):

```
GitHub Actions 每天 17:00 UTC 触发
  → checkout 仓库(取出已有 price_history.csv)
  → 运行 main.py:查 SerpAPI → 更新历史 → 算建议 → 发 Telegram
  → 把今天的价格追加进 CSV,自动 commit & push 回仓库
```

模块拆分(职责单一、可独立测试):

| 模块 | 职责 | 依赖 |
|------|------|------|
| `config.yaml` | 航线、4 对日期、人数、币种、阈值、每天查询组合、决策期切换月份、enabled 开关 | — |
| `flights.py` | 封装 SerpAPI 调用,返回标准化结果(全网最低单价 / 价格档位 / 典型区间 / 最优行程 / 含海航最低单价) | SerpAPI |
| `history.py` | 读写 `data/price_history.csv`;提供历史最低、本周/本月/全部最低、上周/上月滚动均值 | CSV |
| `recommend.py` | **纯函数**:输入今日结果 + 历史 → 输出建议档位 + 徽章 + 理由 | — |
| `notify.py` | 拼 Telegram 消息文本并发送 | Telegram Bot API |
| `main.py` | 编排全流程,控制查询预算与失败处理 | 以上全部 |
| `.github/workflows/daily.yml` | 定时触发 + commit 历史文件;`workflow_dispatch` 手动触发 | — |

### 价格归一化

Google 在选了 3 名乘客时返回的可能是"3 人总价"或"单人价"。`flights.py` 统一归一化为**单人价**;消息中同时显示单人价与 ×3 合计(标注不含行李/税)。具体以实测 SerpAPI 响应为准,实现时确认。

## 4. 数据模型

`data/price_history.csv`,每天每个"城市×日期"组合追加一行:

| 字段 | 说明 |
|------|------|
| `checked_at` | 查询日期 |
| `depart_date` / `return_date` | 行程日期 |
| `destination` | CAN / SZX |
| `currency` | MXN |
| `lowest_overall_pp` | 全网最低单价 |
| `price_level` | low / typical / high(Google) |
| `typical_low` / `typical_high` | Google 典型区间上下沿 |
| `hainan_lowest_pp` | 含海航的最低单价(可空) |
| `best_itinerary` | 最优行程摘要(转机次数 / 航司 / 时长) |

## 5. 建议逻辑(`recommend.py`)

三层信号,每个"城市×日期"组合各算一次:

1. **Google 价格档位**(主信号):偏低 / 正常 / 偏高
2. **相对典型区间**:当前价在 Google 典型区间里偏哪头
3. **相对自建历史**:对比"追踪以来最低 / 近期均价"

输出 5 档建议:

| 建议 | 触发条件 |
|------|----------|
| 🟢🟢 强烈建议买 | 档位=偏低 **且**(价格≤追踪以来最低 或 ≤典型区间下沿) |
| 🟢 可以买 | 档位=偏低(便宜但非新低) |
| 🟡 再观望 | 档位=正常 |
| 🔴 暂不建议 | 档位=偏高 |
| 📉 大跌提醒 | 比近期均价跌 ≥ 8%(可配),叠加在任意档位上 |

**用户最在乎的三个新低节点徽章**(取最高一级显示):

- 🔥 史上新低 > 🟠 本月新低 > 🔵 本周新低

**百分比对比**:消息固定显示 `较上周均价 ±X% · 较上月均价 ±Y%`(由 `history.py` 算滚动均值)。

每天**挑信号最强/最便宜的组合做头条**;若 8 个组合全是正常/偏高,头条显示最便宜组合 + "暂无好价,继续等"。追踪头 ~10 天历史少时,标注"主要参考 Google 信号"。所有阈值放配置。

## 6. 海南航空处理(已决策)

TIJ 出发几乎不可能有纯海航行程(需跨航司拼接),且 Google 的 `include_airlines=HU` 过滤很可能搜出 0 条;独立"海航专查"又会让查询翻倍超额度。

**决策:主看全网最低 + 在结果里标出"含海航最低价"。**

- 每个组合只查 1 次(不加航司过滤)→ 拿全网最低价 + Google 价格信号。
- 在返回的航班列表里扫出**含海航的最低价**一并显示;若无,显示"含海航:暂无可用"。
- 不额外消耗额度。代价:可能漏掉不够便宜的部分海航选项(可接受)。

## 8. 出错处理

核心原则:**没消息 ≠ 没事**。

- SerpAPI 超时/报错:重试最多 3 次(失败不计成功额度);仍失败则跳过该组合,消息标 `⚠️ 部分组合抓取失败`。
- **全部失败**:照发一条 "今日抓取全部失败,请检查",绝不沉默。
- 某组合缺 `price_insights`:回退到仅用历史判断,并标注。
- 结果无海航:显示"含海航:暂无可用"。
- Telegram 发送失败:该次 Actions 运行显示失败(仓库可见)+ 记日志。
- 密钥缺失/无效:立即报错,Actions 日志写清原因。

## 9. 定时与时区

- GitHub Actions cron 用 **UTC**,**不自动适配夏令时**。
- Tijuana = 太平洋时间(夏 UTC-7 / 冬 UTC-8)。
- cron 定 **`0 17 * * *`(17:00 UTC)**:冬令时 = Tijuana 9:00(决策关键期 10–12 月,11/2 后入冬令时,正好落 9 点);夏令时约 10:00。"9 点左右"够用,可在 workflow 调整。
- GitHub 定时任务高负载时可能晚几到十几分钟,属正常。
